🚀 Lunar Lander — Fire the Thruster Against Gravity and Set Down Gently on the Pad Before the Fuel Runs Out

Beat gravity with a short tank of fuel.

Gravity pulls your lander down the whole time — the only thing fighting it is your thruster, and the thruster burns a limited tank of fuel. Hold ↑ or Space to fire the engine out of the lander's belly, and steer by tilting: ← and → rotate the craft, so the thrust pushes in whatever direction the nose is pointing. Your job is to bring it down onto the green landing pad — the one flat ledge in the jagged terrain — gently: slow vertical speed, slow sideways drift, and near-upright. Come in too fast, drift off the pad, or land tilted, and you crash. The read-outs above tell you everything the physics knows: how high you are, how fast you are falling and sliding, how much fuel is left, and your tilt. Land the Eagle and it stays put; misjudge it and it explodes.

This is a real physics simulation, nothing scripted. The lander is just six numbers — a position, a velocity, an angle, and a fuel gauge. Every frame two forces act on it: gravity, which always adds a little downward speed, and thrust, which exists only while you hold the key and still have fuel. The thruster always pushes along the nose, so its force is split into sideways and upward parts with sin and cos of the tilt. Those forces become acceleration, acceleration is added to velocity, and velocity is added to position — the same accel → velocity → position chain that moves every planet and rocket. The ground is a jagged line of points with one deliberately flat pair — that flat pair is your pad. The instant the lander's feet reach the terrain, one test decides everything: were you over the pad, slow enough, and upright enough? All vanilla JavaScript on a canvas, offline, no images, no libraries.

🔨 Build it yourself

    
One tiny state object, two forces, a thrust vector split with sin and cos, the classic accel → velocity → position integration scaled by a clamped dt, a jagged terrain with one flat pad, a straight-line lookup for the ground height, and a single three-part touchdown test. About 120 lines of vanilla JavaScript on a canvas — no images, no libraries. Copy each block in order.

    
    

      
  1. 
        
    1
    The lander is just numbers
    
        
    There is no sprite and no physics engine — the whole spacecraft is one small object. It knows where it is (x, y), how fast it is moving (vx, vy), which way it is tilted (angle, in radians, with 0 meaning nose-up), and how much fuel is left. Everything you see on screen is drawn from these six numbers each frame.
    
        
    const LW = 18, LH = 26;           // lander body size (for drawing + feet)

let L = {
  x: 360, y: 50,                  // position, in pixels
  vx: 0,   vy: 0,                 // velocity, in pixels per second
  angle: 0,                       // tilt in radians · 0 = pointing straight up
  fuel: 100,                      // burns only while the thruster is lit
  thrusting: false
};
    
      
    2. 

      
  2. 
        
    2
    Two forces: gravity and thrust
    
        
    Only two things ever push on the lander. Gravity is constant and always points down, so it is always part of the vertical acceleration. Thrust is the opposite: it exists only when you hold the key and the tank is not empty. Because thrust must be able to cancel gravity and then lift you, its strength is set well above gravity.
    
        
    const GRAVITY = 30;   // px/s², always pulling DOWN (+y)
const THRUST  = 78;   // px/s² out of the engine — must beat gravity
const BURN    = 14;   // fuel spent per second while thrusting

// gravity is unconditional; thrust needs both a held key AND fuel
L.thrusting = keys.thrust && L.fuel > 0;
    
      
    3. 

      
  3. 
        
    3
    Aiming the thrust with sin & cos
    
        
    The engine always fires along the nose, so where the lander points is where the push goes. When the lander is upright (angle 0) the push is straight up. Tilt it, and the same push splits into a sideways part (sin) and an upward part (cos). This one bit of trigonometry is what lets you cancel sideways drift by leaning into it.
    
        
    let ax = 0, ay = GRAVITY;                 // start with gravity on the y axis
if (L.thrusting){
  ax += Math.sin(L.angle) * THRUST;       // sideways part of the push
  ay -= Math.cos(L.angle) * THRUST;       // upward part (negative y = up)
  L.fuel -= BURN * dt;                    // and it costs fuel
}
// angle 0  → sin 0, cos 1 → ax 0, ay = GRAVITY - THRUST  (pure lift)
    
      
    4. 

      
  4. 
        
    4
    Integrate: accel → velocity → position
    
        
    This is the heart of every physics simulation. Acceleration is how fast velocity changes, and velocity is how fast position changes — so each frame you add the acceleration onto the velocity, then add the velocity onto the position, each scaled by dt (the seconds since the last frame). Updating velocity before position is the stable "semi-implicit Euler" method.
    
        
    L.vx += ax * dt;      // acceleration changes velocity
L.vy += ay * dt;
L.x  += L.vx * dt;    // velocity changes position
L.y  += L.vy * dt;

// dt scaling means the flight runs at the same speed on any monitor

  5. 
        
    5
    Rotating the craft
    
        
    Steering is just nudging the angle while a turn key is held, again scaled by dt so it turns at a steady rate. The angle feeds straight back into the thrust split from step 3, so turning and burning are all the control you have. There is no separate "facing" to keep in sync — the single angle value both draws the lander and aims its engine.
    
        
    const ROT = 2.6;                    // radians per second

if (keys.left)  L.angle -= ROT * dt;
if (keys.right) L.angle += ROT * dt;

// that same L.angle is used by Math.sin / Math.cos above AND by ctx.rotate below

  6. 
        
    6
    The terrain and the one flat pad
    
        
    The ground is nothing but a list of points across the width of the canvas at random heights — connect them and you get jagged hills. The landing pad is made simply by forcing two neighbouring points to the same height, which draws as a flat ledge. We remember that pad's left and right edges so the touchdown test can ask "were we over it?".
    
        
    function buildTerrain(){
  const n = 16, seg = W / (n - 1), pts = [];
  for (let i = 0; i < n; i++)
    pts.push({ x: i * seg, y: H - 30 - Math.random() * H * 0.4 });

  const pi = 3 + (Math.random() * (n - 7) | 0);   // where the flat pad sits
  const py = H - 30 - Math.random() * 50;
  pts[pi].y = pts[pi + 1].y = py;                  // two points, one height = flat
  pad = { x1: pts[pi].x, x2: pts[pi + 1].x, y: py };
  return pts;
}
    
      
    7. 

      
  7. 
        
    7
    How high is the ground right here?
    
        
    To know when the lander touches down — and to show altitude — we need the terrain height directly under any x. Find the two points the lander sits between and draw a straight line between them, then read off the y at our x. That linear interpolation turns a handful of points into a continuous surface the lander can rest on.
    
        
    function groundY(x){
  for (let i = 0; i < terrain.length - 1; i++){
    const a = terrain[i], b = terrain[i + 1];
    if (x >= a.x && x <= b.x){
      const t = (x - a.x) / (b.x - a.x);      // 0..1 across this segment
      return a.y + (b.y - a.y) * t;           // straight-line interpolation
    }
  }
  return terrain[terrain.length - 1].y;
}
    
      
    8. 

      
  8. 
        
    8
    Touchdown or crash: three tolerances
    
        
    When the feet reach the ground, one test decides the whole game. You must be over the pad, slow enough in both directions, and near-upright. Miss any one of the three and it is a crash — and the reason is easy to report, because each condition is a separate boolean. These three numbers are the entire difficulty of the game.
    
        
    const SAFE_VY = 28, SAFE_VX = 20, SAFE_ANG = 0.14;   // ~8°

function land(){
  const onPad   = L.x >= pad.x1 && L.x <= pad.x2;
  const gentle  = Math.abs(L.vy) <= SAFE_VY && Math.abs(L.vx) <= SAFE_VX;
  const upright = Math.abs(L.angle) <= SAFE_ANG;
  state = (onPad && gentle && upright) ? "landed" : "crash";
}

  9. 
        
    9
    The loop that ties dt together
    
        
    A single requestAnimationFrame loop drives everything. It measures how many seconds passed since the last frame and clamps it, so a lag spike or a background tab can never let the lander jump straight through the ground. It steps the physics, checks whether the feet have reached the terrain, draws, and asks for the next frame only while still flying.
    
        
    function loop(now){
  const dt = Math.min(0.05, (now - last) / 1000);   // clamp — no teleporting
  last = now;
  if (state === "fly"){
    step(dt);                                        // forces + integration
    if (L.y + LH / 2 >= groundY(L.x)) land();        // did the feet touch?
  }
  draw();
  if (state === "fly") requestAnimationFrame(loop);  // stop when landed/crashed
}

  10. 
        
    10
    Draw the world through the lander's frame
    
        
    Drawing a rotating ship is easy if you move the canvas instead of the ship: translate to the lander's position, rotate by its angle, then draw the body at the origin as if it were never tilted. A flame is drawn below only while thrusting. The HUD simply reads the state numbers — the display never changes the physics, so nothing can drift.
    
        
    function draw(){
  ctx.fillStyle = "#020617"; ctx.fillRect(0, 0, W, H);   // space
  drawTerrain();                                          // hills + green pad
  ctx.save();
  ctx.translate(L.x, L.y);                                // move INTO the lander
  ctx.rotate(L.angle);
  if (L.thrusting) drawFlame();
  drawBody();                                             // drawn upright at (0,0)
  ctx.restore();
}
// HUD: altitude = groundY(L.x) - (L.y + LH/2), speeds/fuel/angle read straight off L
